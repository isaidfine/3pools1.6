# P3 设计需求：数值重构与合成聚焦 (Economy Rebalance)

## 1. 设计目标 (Design Goal)
*   **回归核心玩法**: 确立 **“合成经营 (Merge & Management)”** 为核心体验，而非“抽奖模拟器”。
*   **消灭垃圾时间**: 即使在金币充足的中后期，玩家也必须进行繁琐但有意义的“购买基础素材 -> 背包整理 -> 合成升级”循环，无法通过“钞能力”直接跳过游戏内容。

## 2. 核心改动：概率重构 (Probability Overhaul)

### 2.1 现状分析 (Current State)
*   目前商店直接掉落 Epic (紫色) 的概率约为 **10%**。
*   导致玩家只需约 10 次抽奖（花费 10-50g）即可获得通关道具，完全架空了合成系统。

### 2.2 修改需求 (Requirements)
请工程师修改 `constants.js` 中的概率配置，执行以下 **“断崖式下调”**：

*   **普通池掉率 (Normal Pools)**:
    *   **Common (普通)**: 权重大幅提升 (建议 80%+)
    *   **Uncommon (优秀)**: 适量保留 (建议 15%~19%)
    *   **Rare (稀有)**: 极低概率 (建议 1% 或更低，作为惊喜)
    *   **Epic & Legendary (史诗/传说)**: **直接移除 (0%)** 或 **微乎其微 (0.01%)**。
    *   *设计意图*: 确保玩家只能买到“原材料”。想要高级货？自己合去。

*   **词缀修正 (Affix Adjustments)**:
    *   `purified` (提纯的): 即使是“提纯”池子，最高也只能掉落 **Rare (蓝色)**，绝不能直接掉紫色成品。
    *   `volatile` (波动的): 保持现状（只会掉 Common 或 Legendary），但 Legendary 的概率要控制在 1% 以下，作为超级大奖。

## 3. 预期体验流 (Expected Gameplay Loop)
1.  **接单**: 接到一个要求“紫色手机”的主线任务。
2.  **采购**: 去商店买一大堆“白色手机碎屏”和“白色电池”。
3.  **合成**:
    *   2白 -> 1绿
    *   2绿 -> 1蓝
    *   2蓝 -> 1紫
4.  **压力**: 背包满了！不得不忍痛卖掉一些水果来腾位置。
5.  **通关**: 终于合出来了！(成就感 MAX)

## 4. 交付物 (Deliverables)
*   更新后的 `INITIAL_STAGE_CONFIG` 概率表。
*   (可选) 调整通过 `App.jsx` 开发者工具验证后的 JSON 配置。
